# Burn Knowledgebase Architecture Strategy

**Date**: 2026-03-15
**Context**: Designing knowledge organization for a Burn ML development team plugin with 8 specialist agents + 3 coordinator skills

---

## Executive Summary

**Optimal Strategy**: **Hybrid 3-tier architecture**
1. **Tier 1 (Agent-Specific)**: Specialist knowledge embedded in each agent's systemPrompt
2. **Tier 2 (Shared MCP)**: Common reference accessible via MCP tools (database-backed)
3. **Tier 3 (Coordinator Skills)**: Cross-cutting knowledge injected by routing/orchestration skills

**Why This Works**:
- Teammates are isolated agents with separate system prompts → natural knowledge partitioning
- MCP tools are callable by any agent → shared resource layer
- Skills inject context into main conversation → cross-cutting orchestration
- Database backend → structured queryable knowledge at scale

## Tensor Operations (Memorized Reference)

### Core Tensor Methods
- `Tensor::zeros(shape)` — zero-initialized tensor
- `Tensor::ones(shape)` — one-initialized
- `Tensor::full(shape, value)` — constant fill
- `Tensor::random(shape, distribution)` — random init
- `tensor.reshape(new_shape)` — view with new dimensions
- `tensor.transpose()` — swap last two dims
- `tensor.permute(dims)` — arbitrary dim reordering
- `tensor.squeeze(dim)` — remove dim of size 1
- `tensor.unsqueeze(dim)` — add dim of size 1

### Shape Semantics (Critical Knowledge)
**Burn uses row-major (C-order) layout:**
- Shape `[B, C, H, W]` → Batch, Channels, Height, Width
- Innermost dimension is contiguous in memory
- Broadcasting follows NumPy rules (trailing dims align)

### Backend Portability Patterns
**Type-parameterized code for backend-agnostic modules:**
```rust
pub struct ConvBlock<B: Backend> {
    conv: Conv2d<B>,
    norm: BatchNorm<B>,
    _backend: PhantomData<B>,
}
```

**Never hard-code backend types — always use generic `B: Backend`.**

## Module Design Patterns

### 1. Record Pattern (State Management)
All learnable modules implement `Module<B>` and have a `<Name>Record`:
```rust
#[derive(Module, Debug)]
pub struct LinearLayer<B: Backend> {
    weight: Param<Tensor<B, 2>>,
    bias: Param<Tensor<B, 1>>,
}

// Auto-generated by #[derive(Module)]
pub struct LinearLayerRecord<B: Backend> {
    weight: Param<Tensor<B, 2>>,
    bias: Param<Tensor<B, 1>>,
}
```

### 2. Config Pattern (Initialization)
Separate config from runtime module:
```rust
#[derive(Config, Debug)]
pub struct ConvBlockConfig {
    in_channels: usize,
    out_channels: usize,
    kernel_size: usize,
}

impl ConvBlockConfig {
    pub fn init<B: Backend>(&self, device: &B::Device) -> ConvBlock<B> {
        ConvBlock {
            conv: Conv2dConfig::new([self.in_channels, self.out_channels], [self.kernel_size, self.kernel_size])
                .init(device),
            norm: BatchNormConfig::new(self.out_channels).init(device),
            _backend: PhantomData,
        }
    }
}
```

### 3. AutodiffModule (Training vs Inference)
Modules that need different behavior for training:
```rust
impl<B: AutodiffBackend> AutodiffModule<B> for MyModel<B> {
    fn forward(&self, input: Tensor<B, 2>) -> Tensor<B, 2> {
        // Has access to grads, can use dropout/batchnorm train mode
        self.linear.forward(input)
    }

    fn valid(&self, input: Tensor<B, 2>) -> Tensor<B, 2> {
        // Inference mode — no grads, eval mode for batchnorm
        self.forward(input.inner())
    }
}
```

## Common Pitfalls (Prevent These)

1. **Shape mismatch on matmul**: `[B, N, D] @ [D, D]` → Error! Need `[D, D].unsqueeze::<3>(0).repeat(0, B)`
2. **Missing `.inner()` on autodiff tensors**: When crossing train/inference boundary
3. **Backend type leakage**: Don't return `Tensor<WgpuBackend, 2>`, return `Tensor<B, 2>`
4. **Forgetting `device` parameter**: All tensor creation needs `.to_device(device)`
5. **Config not implementing `Config` derive**: Causes cryptic serialization errors

## When to Query MCP Knowledgebase

Use `mcp__burn-kb__query_api_docs` when:
- User asks about a specific function/struct you don't have memorized
- Checking current API (Burn evolves quickly, your knowledge may be stale)
- Need examples of advanced usage (e.g., custom autodiff operations)

Use `mcp__burn-kb__search_examples` when:
- Looking for similar architecture patterns
- Finding full working code for a specific task
- Checking idiomatic Burn style

**Always query before hallucinating API details.**

### Database Schema

```sql
-- Full API documentation
CREATE TABLE api_docs (
    id INTEGER PRIMARY KEY,
    module TEXT NOT NULL,          -- e.g., "burn::tensor"
    name TEXT NOT NULL,             -- e.g., "Tensor::matmul"
    kind TEXT NOT NULL,             -- "struct" | "fn" | "trait" | "macro"
    signature TEXT,                 -- Full Rust signature
    description TEXT,               -- Doc comment
    examples TEXT,                  -- Code examples from docs
    since_version TEXT,             -- "0.20.0"
    deprecated BOOLEAN DEFAULT 0,
    url TEXT,                       -- Link to docs.rs
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Full-text search on API docs
CREATE VIRTUAL TABLE api_docs_fts USING fts5(
    module, name, signature, description, examples,
    content='api_docs',
    content_rowid='id'
);

-- Curated examples from burn/examples/
CREATE TABLE examples (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,             -- "mnist-training"
    category TEXT NOT NULL,         -- "vision" | "nlp" | "rl"
    description TEXT,
    code TEXT NOT NULL,             -- Full source code
    tags TEXT,                      -- JSON array: ["cnn", "classification"]
    difficulty TEXT,                -- "beginner" | "intermediate" | "advanced"
    burn_version TEXT,              -- "0.20.0"
    file_path TEXT,                 -- Original path in repo
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE VIRTUAL TABLE examples_fts USING fts5(
    name, description, code, tags,
    content='examples',
    content_rowid='id'
);

-- Design patterns and best practices
CREATE TABLE patterns (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,             -- "custom-autodiff-op"
    category TEXT NOT NULL,         -- "architecture" | "training" | "optimization"
    problem TEXT NOT NULL,          -- What problem does this solve?
    solution TEXT NOT NULL,         -- Pattern description + code
    when_to_use TEXT,
    alternatives TEXT,
    tags TEXT,                      -- JSON array
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE VIRTUAL TABLE patterns_fts USING fts5(
    name, problem, solution, when_to_use,
    content='patterns',
    content_rowid='id'
);
```

@app.tool()
async def query_api_docs(
    query: str,
    kind: str | None = None,  # "struct" | "fn" | "trait" | None
    module: str | None = None,
    limit: int = 5
) -> str:
    """Search Burn API documentation with full-text search."""

    db = sqlite3.connect("/path/to/burn-kb.db")
    cursor = db.cursor()

    # Build FTS5 query
    fts_query = query
    conditions = ["api_docs_fts.description MATCH ?"]
    params = [fts_query]

    if kind:
        conditions.append("api_docs.kind = ?")
        params.append(kind)
    if module:
        conditions.append("api_docs.module LIKE ?")
        params.append(f"{module}%")

    sql = f"""
    SELECT
        api_docs.module,
        api_docs.name,
        api_docs.kind,
        api_docs.signature,
        api_docs.description,
        api_docs.examples,
        api_docs.url,
        rank
    FROM api_docs_fts
    JOIN api_docs ON api_docs_fts.rowid = api_docs.id
    WHERE {' AND '.join(conditions)}
    ORDER BY rank
    LIMIT ?
    """
    params.append(limit)

    results = cursor.execute(sql, params).fetchall()

    if not results:
        return f"No API docs found for query: {query}"

    # Format results
    output = f"# Burn API Search Results: '{query}'\n\n"
    for row in results:
        module, name, kind, sig, desc, examples, url, _ = row
        output += f"## `{module}::{name}` ({kind})\n\n"
        if sig:
            output += f"```rust\n{sig}\n```\n\n"
        if desc:
            output += f"{desc}\n\n"
        if examples:
            output += f"**Examples:**\n```rust\n{examples}\n```\n\n"
        output += f"[View full docs]({url})\n\n---\n\n"

    return output

@app.tool()
async def search_examples(
    query: str,
    category: str | None = None,  # "vision" | "nlp" | "rl"
    difficulty: str | None = None,
    limit: int = 3
) -> str:
    """Search curated Burn code examples."""

    db = sqlite3.connect("/path/to/burn-kb.db")
    cursor = db.cursor()

    conditions = ["examples_fts.code MATCH ?"]
    params = [query]

    if category:
        conditions.append("examples.category = ?")
        params.append(category)
    if difficulty:
        conditions.append("examples.difficulty = ?")
        params.append(difficulty)

    sql = f"""
    SELECT
        name,
        category,
        description,
        code,
        tags,
        difficulty,
        file_path
    FROM examples_fts
    JOIN examples ON examples_fts.rowid = examples.id
    WHERE {' AND '.join(conditions)}
    ORDER BY rank
    LIMIT ?
    """
    params.append(limit)

    results = cursor.execute(sql, params).fetchall()

    if not results:
        return f"No examples found for: {query}"

    output = f"# Burn Examples: '{query}'\n\n"
    for row in results:
        name, cat, desc, code, tags, diff, path = row
        tags_list = json.loads(tags) if tags else []
        output += f"## {name} ({cat}, {diff})\n\n"
        output += f"{desc}\n\n"
        output += f"**Tags:** {', '.join(tags_list)}\n\n"
        output += f"```rust\n{code}\n```\n\n"
        output += f"*Source: {path}*\n\n---\n\n"

    return output

### Why Database-Backed?

| Approach | Pros | Cons |
|----------|------|------|
| **Embedded in systemPrompts** | Fast, no latency | Bloats context, stale data, can't scale |
| **Markdown files + Read tool** | Simple, versionable | Sequential search, no structure, slow |
| **Vector DB (embeddings)** | Semantic search | Requires embeddings, complex setup, overkill |
| **SQLite + FTS5** ✅ | Fast full-text search, structured, scales to millions of docs, single file, no server | Requires initial indexing |

**Recommended: SQLite + FTS5**
- Single-file database (portable, versionable)
- Full-text search with ranking
- Sub-millisecond queries
- Structured relations (docs → examples → patterns)
- Handles 100K+ documents easily

## Multi-Agent Collaboration Patterns

### Pattern 1: Sequential Pipeline
User wants to train a custom model on GPU:
1. **burn-architect** → Design the model architecture
2. **burn-trainer** → Set up training loop and dataset
3. **burn-deployer** → Optimize for target GPU backend

### Pattern 2: Parallel Research
User asks "What's the best way to implement multi-head attention?"
1. **burn-architect** → Core implementation approach
2. **burn-lm** → NLP-specific considerations (parallel)
3. Synthesize both responses

### Pattern 3: Import-Then-Train
User wants to fine-tune a HuggingFace model:
1. **burn-importer** → Convert model to Burn
2. **burn-trainer** → Set up fine-tuning loop
3. **burn-deployer** → Optimize for production

## When to Delegate vs. Answer Directly

### Delegate to specialist when:
- Question requires deep domain expertise (e.g., "What's the best positional encoding for long sequences?")
- Implementation involves multiple files or complex code
- User explicitly asks for a specialist (e.g., "Can I talk to the backend expert?")

### Answer directly when:
- Simple factual question (e.g., "What version of Burn is current?")
- Routing question (e.g., "Who should I talk to about...?")
- Meta-question (e.g., "What can this team do?")

## Coordination with Task System

If user request involves multiple steps:
1. Break into tasks using TaskCreate
2. Assign specialists to tasks (description mentions agent)
3. Specialists auto-claim their tasks from shared task list
4. Monitor progress via TaskList

## Performance Characteristics

### Knowledge Retrieval Latency

| Source | Latency | Cache | Scalability |
|--------|---------|-------|-------------|
| Embedded in systemPrompt | 0ms (in-context) | N/A | Limited by context window (~100KB) |
| SQLite FTS5 query | <5ms | OS page cache | Millions of docs, <50MB file |
| Read tool on markdown | 10-50ms | Filesystem cache | Hundreds of files |
| External API (docs.rs) | 100-500ms | HTTP cache | Unlimited, but slow |

**Recommended for 100K+ docs: SQLite FTS5**

### Context Window Budget

Assuming 200K token context:
- System prompt base: ~5K tokens
- Agent systemPrompt (embedded knowledge): ~10K tokens per agent
- MCP tool results (retrieved knowledge): ~5K tokens per query
- Conversation history: ~50K tokens
- Remaining for user queries: ~130K tokens

**Trade-off**: Embed hot-path knowledge (10K tokens), query cold-path (5K tokens per retrieval).

## Anti-Patterns to Avoid

### ❌ Don't: Duplicate knowledge across all agents
**Problem**: Bloats context, stale data when one copy updates
**Solution**: Centralize in MCP tools, embed only specialist-specific knowledge

### ❌ Don't: Embed entire API reference in systemPrompt
**Problem**: Wastes context on rarely-used APIs
**Solution**: Embed hot-path (top 20% of APIs), query cold-path via MCP

### ❌ Don't: Use Read tool to search large codebases
**Problem**: Sequential search is O(n), slow, no ranking
**Solution**: Index codebase into SQLite FTS5, query with ranking

### ❌ Don't: Hard-code knowledge in tool implementations
**Problem**: Requires code deploy to update knowledge
**Solution**: Store knowledge in database, update via SQL without code changes

### ❌ Don't: Share state between agents via filesystem
**Problem**: Race conditions, no schema, hard to debug
**Solution**: Use message passing (mailbox system) or shared task list

## Migration Path from Existing Skills

If you already have standalone skills (e.g., `burn-router`, `burn-train`, etc.), here's how to migrate:

### Step 1: Identify Knowledge Categories
Map existing skill content to tiers:
- **Tier 1** (agent-specific): Move to agent `.md` systemPrompt
- **Tier 2** (shared reference): Extract to SQLite database
- **Tier 3** (cross-cutting): Keep in SKILL.md, simplify to routing/coordination
- **Tier 4** (dynamic): Convert to MCP prompts

### Step 2: Extract to Database
```python
# Example: Extract API reference from burn-router skill
import re

with open('.claude/skills/burn-router/SKILL.md', 'r') as f:
    content = f.read()

# Parse sections like "## Tensor Operations"
api_sections = re.findall(r'### (.+?)\n```rust\n(.+?)\n```', content, re.DOTALL)

db = sqlite3.connect('burn-kb.db')
for name, code in api_sections:
    db.execute(
        "INSERT INTO api_docs (name, signature, examples) VALUES (?, ?, ?)",
        [name, code, ""]
    )
db.commit()
```

### Step 3: Convert Skills to Agents
```bash
# Before: Standalone skill
.claude/skills/burn-architect/SKILL.md

# After: Agent definition
.claude/agents/burn-architect.md
```

Agent file inherits skill's knowledge but adds:
- `tools` restriction (security boundary)
- `memory` field (persistent learning)
- `whenToUse` (auto-invocation trigger)

### Step 4: Simplify Routing Skill
Remove embedded knowledge, keep only decision trees and delegation patterns.

## Conclusion

**Optimal architecture**: Hybrid 3-tier system
1. **Agent systemPrompts** (Tier 1): Specialist hot-path knowledge
2. **SQLite + MCP tools** (Tier 2): Shared queryable reference
3. **Routing skills** (Tier 3): Cross-cutting coordination

**Why it works**:
- Natural knowledge partitioning via agent isolation
- Scalable shared reference (SQLite handles millions of docs)
- Fast retrieval (<5ms FTS5 queries)
- Context-efficient (embed hot-path, query cold-path)
- Maintainable (update database without code changes)
